# 🧩 Token（トークン）とは？

AIが文章を読むとき、**文字や単語をひとまとめにした「かたまり」**に分解します。このかたまりが **Token（トークン）** です。

「Hello, how are you?」を分解すると：

`Hello` `，` `how` `are` `you` `?`

→ **6トークン**　英語は1単語≒1トークンに近い

| 英語の場合 | 日本語の場合 |
|---|---|
| 単語単位で分かれる | 文字や音節で分かれる |
| 1単語 ≒ 1〜2トークン | 1〜2文字 ≒ 1トークン |
| | → **英語より多くなりがち** |

---

# 📩 Input Token と Output Token

| Input Token | | | Output Token |
|---|---|---|---|
| **あなたの質問** | → 🤖 → | | **AIの回答** |
| AIに送る文章のトークン数 | | | AIが返す文章のトークン数 |
| 例：「東京のおすすめランチは？」 | | | 例：ランチの説明文・リストなど |

両方を合計したトークン数が「今回のやりとりの総量」になります。

---

# 💰 Token数と料金の関係

APIを使うとき、**使ったトークン数に応じて料金が決まります。**
送った量（Input）と受け取った量（Output）がそれぞれ課金されます。

| 短い会話（例） | 長い文書分析（例） | 大量処理（例） |
|:---:|:---:|:---:|
| ~500 | ~10,000 | ~100,000 |
| トークン → 安い | トークン → 中程度 | トークン → 高くなる |

イメージ：トークン＝電気の「kWh（使用量）」

- **Input（質問）** は比較的安く課金されます
- **Output（回答）** はやや高く課金されます（生成コストが高いため）
- 長い質問・長い回答 ＝ トークン多い ＝ コスト高い

# 🍎「りんごは赤い」は何トークン？

実際に分解してみましょう：

`りんご` `は` `赤い`

> **約3トークン**です（ツールによって多少異なります）。
>
> 「りんご」「は」「赤い」の3かたまりに分解されます。
> 日本語は英語より細かく分かれるので、同じ意味の文でも英語より多くなることがあります。
>
> 比較：英語の「Apple is red」＝ 3〜4トークンとほぼ同じ

# 🎲 おまけ：なぜ毎回違う回答が出るの？

AIは次のトークンを**確率的に選んで**います。サイコロを振るように、毎回少し違う選択をするため、同じ質問でも回答が変わります。


# 🪟 コンテキストウィンドウとは？

## 一言で言うと

AIが「一度に読める文章の量」に上限があります。
この上限のことを **コンテキストウィンドウ** と呼びます。

---

## 本の「付箋」で考えるイメージ

AIは会話するとき、**これまでのやりとりをすべて毎回読み直して**から返答しています。
ただし、読める量には限界があります。
---

## サイズの目安（Token数）

| モデルの例 | コンテキストウィンドウ | 日本語に換算すると |
|---|---|---|
| 小さめのモデル | 約 4,000 Token | 原稿用紙 約8枚分 |
| Claude など | 約 200,000 Token | 文庫本 約1〜2冊分 |
| GPT-4o など | 約 128,000 Token | 文庫本 約1冊分 |

> 日本語は1〜2文字で1トークンのため、英語より早く上限に近づきます。

---

## なぜ長い会話で精度が下がるのか

### 理由① ウィンドウからあふれる

会話が長くなると、**古いやりとりがウィンドウの外に押し出されて**AIから見えなくなります。
### 理由② 情報が多すぎて薄まる

ウィンドウ内に収まっていても、情報量が多すぎると**重要な部分への注意が分散**します。
人間でも、100ページの資料を一気に渡されると要点を見失うのと同じです。

### 理由③ AIには「記憶」がない

AIはチャットをまたいで記憶を持ちません。
新しいチャットを開いた瞬間、**前回の会話は完全にリセット**されます。

---

## 実務での対処法

### ✅ 対処法①：新しいチャットを始める

話題が変わったら、こまめに新しいチャットを開く。
ウィンドウをリセットすることで、AIが余計な情報に引きずられなくなります。

### ✅ 対処法②：会話を要約して再投入する

長くなってきたら、一度こう聞く：

> 「ここまでの会話を200字で要約してください」

その要約を新しいチャットの冒頭に貼り付けてから続けると、
**文脈を保ちながらウィンドウを節約**できます。

### ✅ 対処法③：最初に条件・前提をまとめて伝える

長い会話が予想される場合は、冒頭に重要な前提をまとめて書いておく。
---

## まとめ

| 項目 | 内容 |
|---|---|
| コンテキストウィンドウ | AIが一度に読める文章量の上限 |
| 精度が下がる理由 | 古い情報が見えなくなる・情報が多すぎて薄まる |
| 対処法 | 新チャット／要約して再投入／冒頭に前提をまとめる |

> 💡 **一番シンプルな指針**：1つのチャットで1つのテーマ。
> 話題が変わったら新しいチャットを開くクセをつけるだけで、精度はかなり安定します。
## 🔍 RAG（Retrieval-Augmented Generation）
外部のデータベースや文書を検索（Retrieve）してから回答を生成する仕組み。
AIが学習時に知らなかった最新情報や社内データを参照できるようになる。
例：社内マニュアルをAIに検索させて、正確な手順を答えさせる。

## 👻 ハルシネーション（Hallucination）
AIが事実ではないことを、あたかも本当のことのように自信満々に答えてしまう現象。
「幻覚」という意味の通り、存在しない論文・人名・URLなどを堂々と生成することがある。
重要な情報は必ず一次ソースで裏取りする習慣が大切。

## 🎯 ファインチューニング（Fine-tuning）
ベースとなるAIモデルに、特定の用途向けのデータを追加学習させてカスタマイズすること。
例：医療専門用語に強いモデル、特定のトーンで話すカスタマーサポートAIなど。
RAGより手間とコストがかかるが、振る舞いそのものを変えられる。

## 🛠 Skills（スキル）
AIが特定のタスクをうまくこなすための「手順書」や「テンプレート」のこと。
Claudeでは、ドキュメント作成・コード生成などの用途ごとにベストプラクティスをまとめたファイルとして使われる。
プロンプトやツールの使い方をパターン化することで、毎回高品質な出力を安定して得られる。

### 🤔 今はまだ関係が薄そう

**ファインチューニング**
モデルそのものをカスタム学習させる話なので、
エンジニアリングチームや大手ベンダーが取り組む領域。
事業開発の立場では「そういうものがある」と知っていれば十分で、
自分で手を動かす場面はほぼない。

**Skills（Claude固有の概念）**
現時点ではClaude APIを直接使って開発する場面向けの話。
既製のHRtechツールを導入・活用する事業開発フェーズでは
直接触れることは少ない。
ただし自社でAI機能を内製するフェーズになれば一気に関係してくる。
